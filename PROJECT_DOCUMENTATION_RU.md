# Документация по проекту NBeauty

## 1. Общее назначение приложения

### Что делает NBeauty

NBeauty — это система для записи к бьюти-мастерам, управления услугами, расписанием, клиентами и салонами. Проект состоит из backend API, Telegram-бота и Telegram Mini App.

Система покрывает две основные стороны:

- клиент ищет мастера, выбирает услугу и записывается;
- мастер ведёт свои записи, услуги, график и базу клиентов.

Дополнительно в проекте есть:

- управление салонами и их ресурсами;
- уведомления через Telegram-бота;
- глобальная админ-панель платформы.

### Какие роли есть

В модели `users.role` есть две основные роли:

- `client` — клиент;
- `master` — мастер.

Отдельно есть ещё два специальных уровня доступа:

- `platform_admins` — глобальный администратор платформы;
- `salon_members.role` — роль пользователя внутри салона: `admin` или `master`.

Важно: глобальный администратор не хранится в `users.role`. Это отдельная таблица.

### Основные сценарии использования

Для клиента:

- авторизоваться через Telegram;
- посмотреть список мастеров;
- найти мастера по имени, специализации или услуге;
- открыть карточку мастера;
- выбрать услугу и свободный слот;
- создать запись;
- посмотреть свои текущие и прошлые записи;
- после завершения записи оставить отзыв.

Для мастера:

- переключиться в режим мастера;
- настроить услуги и рабочие часы;
- смотреть и подтверждать записи;
- создавать запись вручную от имени клиента;
- вести список клиентов;
- добавлять незарегистрированных клиентов;
- хранить заметки по клиентам;
- создавать салон, вступать в салон, управлять участниками и ресурсами.

## 2. Архитектура проекта

### Frontend

Основной frontend находится в `telegram-mini-app/frontend/src`.

Это Next.js-приложение, которое работает как Telegram Mini App. Внутри него реализованы:

- каталог мастеров;
- карточка мастера;
- мастерский dashboard;
- профиль;
- управление услугами;
- рабочие часы;
- мои записи;
- мои клиенты;
- экран салона;
- экран глобального админа.

Frontend общается с backend по HTTP API.

### Backend

Backend реализован на FastAPI и находится в `backend`.

Он отвечает за:

- авторизацию через Telegram;
- профиль пользователя;
- список мастеров;
- создание и управление записями;
- отзывы;
- услуги;
- расписание;
- клиентов мастера;
- салоны и ресурсы;
- глобальную админку.

### База данных

Используется PostgreSQL. Схема БД описана в `backend/migrations/001_initial_schema.sql`.

ORM-модели находятся в `common/models.py`.

### Telegram Mini App

Mini App — основной пользовательский интерфейс. Пользователь открывает приложение из Telegram и работает уже внутри встроенного веб-интерфейса.

Внутри Telegram Mini App авторизация идёт через `initData`, которое Telegram передаёт приложению.

### Telegram-бот

Бот находится в папке `bot`.

Он нужен для:

- входа в приложение через кнопку WebApp;
- dev/browser fallback-авторизации через login-session;
- уведомлений мастеру о новой записи;
- уведомлений клиенту о подтверждении и отмене записи;
- запроса отзыва после завершённой записи;
- обработки inline-кнопок подтверждения, отмены и оценки.

### Как части взаимодействуют друг с другом

Схема взаимодействия такая:

1. Пользователь открывает Mini App из Telegram.
2. Frontend получает `Telegram.WebApp.initData`.
3. Frontend отправляет `initData` в backend.
4. Backend проверяет подпись Telegram и создаёт/обновляет пользователя.
5. Backend выдаёт JWT.
6. Frontend работает с API по JWT.
7. При создании или изменении записи backend запускает уведомления через Telegram-бота.
8. Бот отправляет сообщения мастеру и клиенту.
9. После завершения записи бот может запросить отзыв.

## 4. Авторизация

### Как происходит вход через Telegram

Основной сценарий входа:

1. Telegram открывает Mini App и передаёт `window.Telegram.WebApp.initData`.
2. Frontend отправляет `initData` в `POST /api/auth/telegram`.
3. Backend в `validate_telegram_init_data()`:
   - разбирает строку;
   - проверяет `hash`;
   - проверяет свежесть `auth_date`;
   - извлекает объект `user`.
4. После проверки backend вызывает `upsert_telegram_user()`.
5. Backend создаёт JWT и возвращает его фронтенду.

### Что такое `tg_id`

`tg_id` — это постоянный Telegram ID пользователя.

В проекте он является главным идентификатором пользователя:

- это primary key таблицы `users`;
- он попадает в JWT как `sub`;
- по нему связываются записи, салоны, услуги, заметки и другие сущности.

### Как создаётся или обновляется пользователь

Логика находится в `common/telegram_users.py`.

При первом входе:

- создаётся запись в `users`;
- роль ставится `client`;
- сохраняются имя, username, аватар и телефон, если они пришли.

При повторном входе:

- пользователь ищется по `tg_id`;
- `full_name` не затирается телеграм-данными, если уже был изменён внутри приложения;
- `username` обновляется только если не нарушает `UNIQUE`;
- `avatar` и `telephone_number` дополняются, если раньше были пустыми.

### Как хранится роль пользователя

Основная роль хранится в `users.role`:

- `client`;
- `master`.

Глобальный админ хранится отдельно в `platform_admins`.

Роль в салоне хранится отдельно в `salon_members.role`.

## 5. Роли

### Клиент

Клиент видит клиентский workflow:

- каталог мастеров;
- карточку мастера;
- мастер записи;
- свои записи;
- профиль.

### Мастер

Мастер видит мастерский workflow:

- dashboard записей;
- услуги;
- рабочие часы;
- профиль;
- салоны;
- мои клиенты;
- возможность ручного создания записи.

### Чем отличаются workflow

Разница не только в интерфейсе, но и в доступных действиях:

- клиент записывается к мастеру;
- мастер принимает и управляет записями;
- мастер может создавать запись не только для зарегистрированного клиента, но и для guest client;
- мастер ведёт собственную клиентскую базу и заметки;
- мастер может работать с салонами и ресурсами.

Во frontend отдельно различаются:

- `role` — кем является аккаунт;
- `viewMode` — какой workflow сейчас открыт.

То есть мастер может временно переключиться в клиентский режим, не теряя роль `master`.

## 6. Клиентский workflow

### Просмотр мастеров

Клиент открывает экран `DiscoveryScreen`.

Frontend вызывает `GET /api/masters/` и получает список мастеров с:

- именем;
- специализацией;
- рейтингом;
- услугами;
- отзывами;
- расписанием.

### Поиск мастеров

Поиск реализован на frontend по уже загруженному списку мастеров.

Фильтрация идёт по:

- имени мастера;
- специализации;
- названиям услуг.

Отдельного backend endpoint для поиска мастеров в текущем коде нет.

### Просмотр карточки мастера

Карточка мастера открывается в модальном окне.

В ней показываются:

- имя;
- username;
- специализация;
- рейтинг;
- количество отзывов;
- список отзывов.

Есть кнопка записи и шаринга профиля через Telegram deep link.

### Выбор услуги

В `BookingWizard` клиент сначала выбирает услугу мастера.

### Создание записи

Дальше клиент:

1. выбирает дату;
2. frontend загружает слоты через `GET /api/appointments/{master_id}/slots`;
3. клиент выбирает время;
4. frontend отправляет `POST /api/bookings/create` с `clientId=current_user.tg_id`.

После создания записи мастеру уходит уведомление в Telegram.

### Просмотр своих записей

Экран `MyBookingsScreen` показывает:

- активные записи через `GET /api/appointments/my?role=client`;
- историю через `GET /api/appointments/my/history?role=client`.

Клиент может отменить активную запись.

### Отзыв после завершённой записи

После завершения записи backend автоматически переводит её в `completed`.

Если запись принадлежит зарегистрированному клиенту (`client_id != null`) и отзыва ещё нет, бот отправляет запрос на оценку.

Отзыв можно оставить:

- через backend API `POST /api/reviews/`;
- через Telegram-бота по inline-кнопкам.

## 7. Workflow мастера

### Просмотр своих записей

Мастерский dashboard показывает записи мастера на выбранную дату.

Источники данных:

- `GET /api/appointments/my?role=master`;
- `GET /api/appointments/my/history?role=master`.

Мастер может:

- подтвердить запись;
- отменить запись;
- добавить новую запись вручную.

### Управление услугами

Экран `ServiceManagement` работает с `api/services`.

Мастер может:

- посмотреть свои услуги;
- создать услугу;
- изменить название, цену, длительность;
- привязать услугу к ресурсу салона;
- удалить услугу, если нет связанных записей.

### Профиль мастера

В профиле мастер может:

- переключать workflow master/client;
- редактировать имя, аватар, телефон, специальность;
- создавать салон;
- вступать в салон по invite code;
- открывать экран салона;
- открывать экран “Мои клиенты”;
- открывать экран рабочего времени.

### Раздел “Мои клиенты”

Мастер имеет отдельный раздел клиентов:

- список клиентов;
- поиск по имени и телефону;
- карточка клиента;
- история записей;
- заметка о клиенте;
- создание guest client;
- удаление guest client.

### Зарегистрированные и незарегистрированные клиенты

Зарегистрированный клиент:

- это пользователь из `users`;
- у него есть `tg_id`;
- записи к мастеру содержат `client_id`.

Незарегистрированный клиент:

- это запись в `guest_clients`;
- у него нет Telegram-аккаунта внутри системы;
- записи к мастеру содержат `guest_client_id`.

### Заметки мастера о клиентах

Заметки хранятся в `client_notes`.

Они привязаны:

- либо к `client_id`,
- либо к `guest_client_id`.

Одна заметка принадлежит одному мастеру и одному клиенту его базы.

## 8. База данных

### Основные таблицы и их назначение

#### `users`

Основная таблица пользователей Telegram.

Содержит:

- `tg_id`;
- имя;
- username;
- роль `client/master`;
- аватар;
- специальность;
- телефон;
- рейтинг мастера.

#### `services`

Услуги мастеров.

Хранит:

- мастера;
- салон;
- ресурс;
- название;
- длительность;
- цену;
- активность.

#### `appointments`

Записи к мастерам.

Содержит:

- мастера;
- зарегистрированного клиента или guest client;
- услугу;
- салон;
- ресурс;
- время начала и конца;
- статус.

#### `reviews`

Отзывы по завершённым записям.

Один отзыв связан ровно с одной записью через `appointment_id`.

#### `guest_clients`

Незарегистрированные клиенты мастера.

Используются для ручных записей, когда у клиента нет аккаунта внутри системы.

#### `client_notes`

Личные заметки мастера о клиенте.

Заметка может относиться либо к зарегистрированному клиенту, либо к guest client.

#### `salons`

Салоны или студии.

Содержат владельца, имя и `invite_code`.

#### `salon_members`

Связь пользователей с салонами.

Хранит роль пользователя внутри салона:

- `admin`;
- `master`.

#### `resources`

Ресурсы салона:

- кабинет;
- кресло;
- оборудование и т.д.

Их можно резервировать вместе с записью.

#### `master_schedules`

Базовое недельное расписание мастера.

Хранит:

- день недели;
- включён ли день;
- время начала и конца;
- опционально салон.

#### `telegram_login_sessions`

Служебные одноразовые сессии входа через бота.

Используются для browser/dev fallback, когда приложение открывают не внутри Telegram Mini App.

### Связи между таблицами

- `users -> services` : один мастер имеет много услуг;
- `users -> appointments` : один мастер имеет много записей;
- `users -> appointments` : один зарегистрированный клиент имеет много записей;
- `appointments -> reviews` : одна запись имеет максимум один отзыв;
- `users -> salons` : пользователь может владеть салонами;
- `salons -> salon_members` : в салоне много участников;
- `salons -> resources` : у салона много ресурсов;
- `salons -> services` : услуга может быть привязана к салону;
- `resources -> services` : услуга может требовать конкретный ресурс;
- `resources -> appointments` : запись может занимать ресурс;
- `users -> guest_clients` : guest clients принадлежат конкретному мастеру;
- `users/guest_clients -> client_notes` : заметка относится либо к registered client, либо к guest client;
- `users -> master_schedules` : у мастера есть недельное расписание;
- `users -> telegram_login_sessions` : login-session после завершения связывается с пользователем.

## 9. Логика записей

### Как создаётся запись клиентом

Клиент создаёт запись на frontend через `BookingWizard`.

Backend в `POST /api/bookings/create`:

1. загружает услугу;
2. проверяет, что услуга принадлежит выбранному мастеру;
3. вычисляет `end_time` по длительности услуги;
4. создаёт дефолтное расписание мастера, если его ещё нет;
5. проверяет, что мастер работает в этот день и в это время;
6. проверяет конфликты по мастеру и по ресурсу;
7. создаёт запись со статусом `pending`;
8. отправляет уведомление мастеру.

### Как создаётся запись мастером

Мастер может вручную создать запись в `AddBookingDrawer`.

Там есть два сценария:

- выбрать зарегистрированного клиента;
- создать или выбрать guest client.

Для мастера backend разрешает создавать запись:

- зарегистрированному клиенту по `client_id`;
- незарегистрированному клиенту по `guest_client_id`.

### Чем отличается registered client от guest client

`registered client`:

- есть в `users`;
- имеет `tg_id`;
- может сам авторизоваться;
- получает уведомления и может оставлять отзыв.

`guest client`:

- хранится в `guest_clients`;
- создаётся мастером вручную;
- не имеет Telegram-авторизации в системе;
- не должен получать сценарий отзыва.

### Почему `client_id` nullable

Потому что запись может принадлежать не только зарегистрированному клиенту, но и guest client.

Если запись создана для незарегистрированного клиента, поле `client_id` остаётся `NULL`.

### Зачем `guest_client_id`

`guest_client_id` нужен, чтобы хранить связь записи с незарегистрированным клиентом мастера.

Без этого мастер не смог бы:

- записывать клиентов без аккаунта;
- хранить историю их визитов;
- вести заметки по ним.

### Как предотвращаются пересечения времени

Защита сделана в двух слоях.

#### На уровне backend-логики

При создании записи backend проверяет пересечение:

- по мастеру;
- по ресурсу, если услуга требует ресурс.

Если найден конфликт, запись не создаётся.

#### На уровне базы данных

В таблице `appointments` есть exclusion constraints:

- `ex_appointments_master_time_overlap` — нельзя пересекать время одного мастера;
- `ex_appointments_resource_time_overlap` — нельзя пересекать время одного ресурса.

Это защищает даже в случае конкурентных запросов.

## 10. Логика отзывов

### Когда можно оставить отзыв

Отзыв можно оставить только если:

- запись существует;
- запись принадлежит текущему клиенту;
- статус записи `completed`;
- отзыва ещё нет.

Эта проверка есть в `POST /api/reviews/`.

### Почему guest client не должен запускать отзыв

В коде это явно учтено.

Запрос на отзыв отправляется только если:

- запись завершена;
- `appointment.client_id is not None`.

То есть guest client не получает сценарий отзыва, потому что:

- он не является зарегистрированным пользователем;
- у него нет Telegram ID в `users`;
- ему нельзя выдать персональное действие по отзыву.

### Как пересчитывается рейтинг мастера

Рейтинг пересчитывается на уровне БД.

В миграции есть функция `recalculate_master_rating(p_master_id)` и trigger `trigger_reviews_recalculate_rating`.

При вставке, обновлении или удалении отзыва:

1. находится мастер по записи;
2. считается средний рейтинг по всем отзывам этого мастера;
3. значение записывается в `users.rating`.

## 13. Что можно рассказать на защите

### Краткий текст на 2–3 минуты

NBeauty — это Telegram Mini App для записи к бьюти-мастерам и управления их работой. Клиентская часть позволяет авторизоваться через Telegram, найти мастера, открыть его карточку, выбрать услугу, свободный слот и записаться. Мастерская часть позволяет мастеру управлять своими услугами, графиком, клиентами и записями, а также вести салон и его ресурсы.

Архитектурно проект разделён на три части: frontend на Next.js, backend на FastAPI и Telegram-бот на aiogram. Frontend работает как Mini App и общается с backend через REST API. Backend хранит бизнес-логику, проверяет Telegram-авторизацию, создаёт JWT, работает с PostgreSQL и запускает уведомления через бота. Бот используется не только для открытия приложения, но и для подтверждения записей, уведомлений и запроса отзывов.

В базе данных есть отдельные сущности для пользователей, услуг, записей, отзывов, салонов, ресурсов и двух типов клиентов: зарегистрированных и незарегистрированных. Это позволило реализовать два сценария работы мастера: запись обычных пользователей системы и ручное ведение клиентов, у которых ещё нет аккаунта в приложении. Для защиты от конфликтов времени проверка сделана и в backend, и на уровне PostgreSQL через exclusion constraints.

### Ключевые технические решения

- авторизация через Telegram `initData` с проверкой подписи;
- `tg_id` как главный идентификатор пользователя;
- разделение ролей `client` и `master`;
- отдельная таблица `platform_admins` вместо хранения глобального админа в `users.role`;
- отдельная сущность `guest_clients` для незарегистрированных клиентов мастера;
- отдельная сущность `client_notes` для заметок;
- автоматическое завершение записей после `end_time`;
- автоматический пересчёт рейтинга мастера через trigger в БД;
- защита от пересечения времени и по мастеру, и по ресурсу;
- Telegram-бот как часть бизнес-процесса, а не только входная точка.

### Возможные вопросы преподавателя и краткие ответы

**Почему выбран Telegram Mini App?**  
Потому что целевая аудитория уже находится в Telegram, и это позволяет использовать готовую авторизацию и бот-уведомления.

**Почему `tg_id` — primary key?**  
Потому что текущая модель входа полностью завязана на Telegram, и это самый стабильный идентификатор пользователя.

**Зачем нужны `guest_clients`?**  
Чтобы мастер мог вести записи и историю клиентов, которые ещё не зарегистрированы в системе.

**Почему `client_id` nullable?**  
Потому что запись может относиться либо к зарегистрированному клиенту, либо к guest client.

**Как предотвращаются двойные бронирования?**  
Через backend-проверки конфликтов и через exclusion constraints PostgreSQL.

**Как считается рейтинг мастера?**  
Среднее значение отзывов по его завершённым записям, пересчитывается триггером БД.

**Почему guest client не получает отзыв?**  
Потому что у него нет аккаунта пользователя и Telegram ID в таблице `users`, а логика отзывов привязана к зарегистрированному клиенту.

**Можно ли мастеру переключиться в клиента?**  
Да. У мастера есть переключение workflow без потери роли `master`.

**Есть ли в проекте салонная логика?**  
Да. Есть салоны, участники салона, invite code и ресурсы салона.

**Что реализовано частично?**  
Endpoint `complete` есть, но фактическое завершение записи выполняется автоматически, а не кнопкой мастера. Также frontend работает как single-screen state machine, а не как полноценный многостраничный веб-сайт.

## Честные замечания по текущей реализации

- Основной рабочий frontend — это `telegram-mini-app/frontend/src`. Папка `web-app/frontend` в текущем репозитории не выглядит как живая часть приложения и в основном содержит артефакты.
- Endpoint `PUT /api/appointments/{appointment_id}/complete` не завершает запись вручную. Он только возвращает запись, если она уже завершилась автоматически по времени.
- Поиск мастеров сделан на frontend по загруженному списку, а не отдельным серверным поиском.
- Login-session через `telegram_login_sessions` нужен в первую очередь для запуска приложения в обычном браузере во время разработки, а не для основного сценария внутри Mini App.
- В проекте нет отдельной финансовой логики оплаты. Поле `price` используется для отображения услуги.
